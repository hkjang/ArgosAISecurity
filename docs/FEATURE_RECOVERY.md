# Argos 백업 및 복구 기능 분석서

**Argos 백업 및 복구 엔진**은 랜섬웨어 침해 또는 악성 변조로 인해 파일 데이터가 훼손되었을 때, 변경 이전 상태의 원본 데이터를 완벽한 무결성 검증을 거쳐 원자적으로 안전 복원해 주는 재해 복구 솔루션입니다.

---

## 1. 핵심 설계 및 컴포넌트

백업 및 복구는 [argos-recovery](file:///d:/project/ArgosAISecurity/crates/argos-recovery/src) 라이브러리가 전담합니다.
- **`BackupStore`** ([lib.rs](file:///d:/project/ArgosAISecurity/crates/argos-recovery/src/lib.rs)): CAS(Content-Addressed Storage) 디렉터리 제어 및 인덱스 SQLite DB(`index.db`) 인터페이스를 관리하며 백업, 복구, 정리(`prune`) 프로세스를 수행하는 핵심 코어 클래스입니다.
- **`BackupVersion`**: 백업 인덱스 DB 상의 개별 파일 버전을 표현하는 메타데이터로, 버전 번호, 해시값, 원본 경로, 파일 크기, 백업 타임스탬프, 유발 PID를 보관합니다.

---

## 2. 내용 주소 지정 백업 (CAS) 메커니즘

Argos는 데이터의 불필요한 중복 적재를 차단하여 백업 스토리지 용량을 절약하기 위해 **내용 주소 지정 저장소(Content-Addressed Storage, CAS)** 아키텍처를 채택했습니다.

### 2.1. 데이터 저장 방식
- 파일 내용(데이터 바이트 스트림)의 SHA-256 해시값을 고유 키로 활용하여 저장 디렉터리를 구성합니다:
  - 경로 구조: `<backup_dir>/objects/<해시앞2자리>/<해시>`
  - 예: 해시가 `5a283de...` 인 경우, `objects/5a/5a283de...` 파일로 저장됩니다.
- 디렉터리 내 수만 개의 객체가 쌓였을 때 단일 폴더 아래 파일 한도 초과 및 탐색 속도 저하가 일어나는 현상을 막기 위해 해시 앞 2자리를 이용한 샤딩 분할 처리를 적용했습니다.

### 2.2. 중복 제거 백업 프로세스 (`backup`)
1. 변경 이벤트를 수신한 파일 경로를 타겟으로 지정합니다.
2. 파일 크기가 정책상 허용된 최대 단일 백업 크기(`max_file_bytes`, 기본값 50MB)를 초과하는 대용량 파일인 경우 백업 대상에서 즉각 제외해 디스크 풀(Full) 에러 리스크를 차단합니다.
3. 데이터 전체를 임시 버퍼로 로드하여 SHA-256 해시 키를 획득합니다.
4. SQLite 인덱스 DB(`index.db` 내 `versions` 테이블)를 쿼리해 해당 경로의 직전 등록 버전 해시를 가져와 비교합니다.
5. 직전 백업 버전 해시와 현 실측 해시가 동일한 경우, 불필요한 중복 쓰기 I/O를 막기 위해 백업 요청을 스킵합니다.
6. CAS 영역에 해당 해시 객체 파일이 존재하지 않는 경우에 한하여 임시 파일(`.tmp`)을 생성해 바이너리 스트림을 다 쓰고, rename 시스템 콜을 실행해 원자적으로 대치 저장함으로써 부분 쓰기 장애를 예방합니다.
7. 인덱스 DB `versions` 테이블에 신규 버전 메타데이터를 기록합니다.

## 3. 해시 무결성 검증 기반 원자적 복구 (`restore`)

관리자가 CLI 도구를 구동해 특정 시점으로 파일의 원복 명령(`argos restore <경로> --before-ms <시각>`)을 시동하면 다음 단계로 처리됩니다.

```
[복구 요청 접수] ──► SQLite index.db 조회 ──► 시점 이전 최적의 BackupVersion 획득
                                                          │
   ┌──────────────────────────────────────────────────────┘
   ▼
[CAS objects/ 경로에서 파일 로드] ──► 바이너리 로드 및 실시간 SHA-256 재검산
                                                          │
   ├─► [해시 불일치 (변조/오염)] ──► IntegrityMismatch 에러 발생 및 즉시 중단
   └─► [해시 일치 (무결성 통과)]
                               │
                               ▼
[임시 복원 파일 작성] ➔ <원본경로>.argos-restore-tmp 파일 생성 및 쓰기
                               │
                               ▼
[원자적 Rename 대치] ➔ rename(tmp, 원본경로) 시스콜 실행 및 파일 원복 완료
```

1. **시점 쿼리**: `index.db` 내 `versions` 테이블에서 타겟 타임스탬프(`before_ms`) 이전이면서 가장 높은 ID 번호의 `BackupVersion` 정보를 도출합니다.
2. **해시 무결성 검증**: CAS 영역에서 해시 객체를 읽어 들여 SHA-256 해시를 즉석에서 직접 다시 계산합니다. 이 값이 `BackupVersion`에 박힌 검증 해시 키와 다르면 데이터 위변조 공격이나 백업 파일 유실 사고로 판단해 복구 작업을 즉각 오류로 거부하고 차단합니다.
3. **원자적 쓰기 교체**: 복구 쓰기 도중 발생하는 파일 경합을 우회하기 위해 대상과 동일 경로 상에 임시 파일(`<원본경로>.argos-restore-tmp`)을 우선 생성해 쓴 뒤, OS 시스콜인 `rename` 함수를 호출해 원본을 순간 대치 원복함으로써 무결성 보장을 매끄럽게 달성합니다.

## 4. 백업 보존 정책 및 노후 데이터 정리 (`prune`)

백업 파일이 서버 디스크를 완전히 채워버리는 리스크에 대처하기 위해 주기적으로 정리 작업을 처리합니다.

- **버전 수 Pruning**:
  - `versions` 데이터베이스에서 개별 경로 단위로 최근 버전 이력 수량(`keep_versions`, 기본값 5개)을 카운팅하여 오래된 레코드들을 인덱스 DB에서 완전히 지웁니다.
- **가비지 컬렉션(참조 해제 객체 소거)**:
  - 인덱스 DB에서 잔존하고 있는 고유 해시 목록(`SELECT DISTINCT hash FROM versions`)을 `live` 해시 셋으로 집계합니다.
  - CAS 디렉터리 하위의 모든 객체 파일을 순회하여 해당 파일명이 `live` 셋에 포함되지 않는 경우(인덱스 상에서 더 이상 참조하지 않는 무효화된 백업 파일) 디렉터리에서 지우는 가비지 컬렉션을 수행합니다.
